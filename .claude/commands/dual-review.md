# /dual-review - Opus & Sonnet 병렬 검토 스킬

대상 파일: $ARGUMENTS (기본값: plan.md)

## 실행 절차

대상 MD 파일을 6개 에이전트가 병렬로 검토한다. 각 에이전트는 서로 다른 관점에서 분석한다.

### Step 1: 대상 파일 읽기

먼저 Read 도구로 대상 파일의 현재 내용을 읽는다. $ARGUMENTS가 비어있으면 `plan.md`를 대상으로 한다.

### Step 2: 6개 에이전트 병렬 실행

아래 6개 Agent를 **반드시 하나의 메시지에서 동시에** 병렬 실행한다. 각 에이전트에게 파일 전체 내용을 전달하고, 해당 관점에서 검토하도록 지시한다.

**Opus 에이전트 3개** (model: opus):

1. **Opus-구조 검토**: 문서 구조, 논리적 흐름, 섹션 누락 여부, 중복 제거 관점에서 검토. 구체적인 수정 제안을 old_text → new_text 형태로 제시.

2. **Opus-기술 검토**: 기술적 정확성, API 사용법, 타입 정의, 아키텍처 결정의 타당성 검토. 잘못된 부분이나 개선점을 old_text → new_text 형태로 제시.

3. **Opus-완성도 검토**: 엣지 케이스, 누락된 고려사항, 실현 가능성, 모호한 표현 검토. 보완이 필요한 부분을 old_text → new_text 형태로 제시.

**Sonnet 에이전트 3개** (model: sonnet):

4. **Sonnet-실용성 검토**: 구현 난이도, 우선순위 적절성, 단계별 의존성, 현실적 실행 가능성 검토. 수정 제안을 old_text → new_text 형태로 제시.

5. **Sonnet-일관성 검토**: 용어 통일, 네이밍 컨벤션, 파일 경로와 설명의 일치 여부, 오타 검토. 수정 제안을 old_text → new_text 형태로 제시.

6. **Sonnet-UX 검토**: 사용자 경험 관점에서 UI 흐름, 반응형 설계, 에러 처리, 접근성 검토. 수정 제안을 old_text → new_text 형태로 제시.

각 에이전트 프롬프트에 반드시 포함할 것:
- 대상 파일의 전체 내용
- "읽기 전용 검토이다. 파일을 직접 수정하지 말고, 검토 결과를 텍스트로 반환하라."
- "수정 제안은 반드시 `[수정]`, 의견은 `[의견]`, 긍정 평가는 `[OK]` 태그를 붙여라."
- "200자 이내로 간결하게 각 항목을 작성하라."

### Step 3: 결과 종합

6개 에이전트의 결과를 수집한 후:

1. 모든 `[수정]` 항목을 목록으로 정리한다
2. 2개 이상의 에이전트가 동일/유사한 수정을 제안한 항목은 **채택 (높은 합의)**로 표시
3. 1개 에이전트만 제안한 항목은 **검토 필요**로 표시
4. `[의견]` 항목은 참고사항으로 별도 정리

### Step 4: 사용자에게 결과 보고

아래 형식으로 사용자에게 보고한다:

```
## Dual Review 결과 (대상: {파일명})

### 채택 (높은 합의) - 자동 적용 예정
| # | 수정 내용 | 제안자 |
|---|----------|--------|
| 1 | ...      | Opus-구조, Sonnet-일관성 |

### 검토 필요 - 사용자 확인 후 적용
| # | 수정 내용 | 제안자 | 이유 |
|---|----------|--------|------|
| 1 | ...      | Opus-기술 | ... |

### 참고 의견
- ...
```

### Step 5: 사용자 승인 후 적용

AskUserQuestion으로 사용자에게 확인:
- "채택 항목 전체 적용" (Recommended)
- "항목별 개별 선택"
- "적용하지 않음"

승인 후 Edit 도구로 대상 MD 파일에 수정사항을 적용한다.
